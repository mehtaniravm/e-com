The Core Concept — What "Autonomous" Means Here
Understanding how self-reasoning agents differ from chatbots
💡
Key Insight: A normal Claude chat is reactive — you prompt, it responds. An autonomous agent is proactive — it has a goal, a state machine, tools, and a reasoning loop. It decides what to do next on its own, only interrupting you when it genuinely can't proceed alone.
❌ What You Have Now (Manual)
  • You open Claude, write a prompt, read the output
  • You copy output, paste it into the next agent
  • You manually check: "did the tests pass?"
  • You decide when to move from stories → architecture
  • If you forget a step, it's skipped
  • Each conversation has no memory of the last
✅ What You Want (Autonomous)
  • You drop in a Jira Epic URL → pipeline starts
  • Agents hand off to each other automatically
  • Each agent reads the previous agent's output
  • Tests run, fail → agent retries and fixes code
  • Pipeline only pauses at real decision points
  • Full audit log of every decision made
🔄 The Reasoning Loop (How Each Agent Thinks)

Every autonomous agent follows this internal loop on every cycle:

OBSERVE  → Read current state (files, test results, Jira status, previous agent output)
ORIENT   → What phase am I in? What's my goal? What's blocking me?
DECIDE   → Can I proceed autonomously OR do I need human input?
ACT      → Execute tools (write code, run tests, update Jira, call next agent)
RECORD   → Write decision + rationale to pipeline state file
REPEAT   → Loop back to OBSERVE unless phase is COMPLETE or BLOCKED

This is the ReAct pattern (Reasoning + Acting) — the foundation of all autonomous AI agents.

The Autonomous Pipeline — Phase by Phase
Exactly what runs automatically vs what needs you
AUTO — agent proceeds alone
PAUSE — agent waits for your input
GATE — hard stop, needs explicit approval
GATE
~2 min
📥 Phase 0 — Epic Intake

You provide: Jira Epic URL or pasted Epic text. That's the only manual trigger in the entire pipeline. Everything else flows from here.

HUMAN: provide epic AUTO: validate epic has enough detail PAUSE: ask clarifying questions if epic is vague
AUTO
~5 min
📝 Phase 1 — Story Generation

Story Agent reads the epic, generates all user stories with story points, acceptance criteria, and service assignments. Writes output to pipeline-state.json and creates Jira tickets automatically.

AUTO: read epic → generate stories AUTO: estimate story points (Fibonacci) AUTO: create Jira stories via MCP PAUSE: if story count > 30 or epic spans > 3 services
PAUSE
~10 min
✅ Gate 1 — Story Approval

Agent sends you a summary: "Generated 14 stories, 47 story points. Here's the breakdown. Reply APPROVE to continue, or tell me what to change."

HUMAN: approve or modify stories AUTO: if no response in 30min, resend reminder
AUTO
~8 min
🏛 Phase 2 — Architecture Design

Architect Agent reads approved stories, generates service boundaries, API contracts, data models, and security design. Checks existing codebase for conflicts. Saves to ARCHITECTURE.md.

AUTO: generate architecture from stories AUTO: check for breaking changes vs existing code PAUSE: if new external service dependency detected PAUSE: if breaking API change found
PAUSE
~15 min
✅ Gate 2 — Architecture Approval

Agent sends architecture summary with any flagged risks. This is important — architecture decisions are expensive to reverse. Human reviews and approves or redirects.

HUMAN: approve architecture + risk decisions
AUTO
~5 min
📅 Phase 3 — Sprint Planning

Planner Agent reads stories + architecture, generates sprint plan respecting dependencies. Assigns stories to sprints, creates GitHub Project board tasks.

AUTO: generate sprint breakdown AUTO: validate no circular dependencies AUTO: create GitHub issues per story
AUTO
~60-120 min
💻 Phase 4 — Implementation (Claude Code)

Dev Agent runs Claude Code in a loop. For each story: generate code → run tests → if fail, fix and retry (up to 3 times) → if still failing, escalate to human. Creates PRs automatically.

AUTO: implement story by story in dependency order AUTO: run mvn test after each story AUTO: retry on test failure (max 3 attempts) PAUSE: after 3 failures → ask human for direction AUTO: create feature branch + PR per story
AUTO
~20 min
🧪 Phase 5 — Testing

Test Agent writes JUnit tests for all new code, runs TestContainers integration tests, runs Playwright E2E suite. Reports coverage. If coverage < 80%, writes more tests automatically.

AUTO: generate tests for all new classes AUTO: run full test suite AUTO: check coverage threshold (80%) AUTO: add tests if coverage below threshold PAUSE: if E2E tests fail after 2 retries
GATE
~10 min
✅ Gate 3 — Deployment Approval

Hard stop before any cloud deployment. Agent sends: test results, coverage report, PR links, and deployment plan. Requires explicit DEPLOY approval. This cannot be bypassed.

HUMAN: explicit deploy approval required
AUTO
~15 min
🚀 Phase 6 — Build & Deploy

DevOps Agent builds Docker images, pushes to Artifact Registry, applies K8s manifests to GKE, runs smoke tests, reports deployment URL.

AUTO: docker build + push AUTO: kubectl apply manifests AUTO: run smoke tests against live URL PAUSE: if smoke tests fail → rollback + notify
⏱
Total Pipeline Time: Epic → Deployed app in approximately 3–4 hours autonomous runtime with only 3 human interactions (Story Approval, Architecture Approval, Deploy Approval). Your actual clock time is maybe 30 minutes of reading and clicking.

The Orchestrator — Actual Code
This is the engine. A Node.js script you run once and it drives everything.
📁 File: orchestrator/index.js
// SDLC Pipeline Orchestrator
// Run: node orchestrator/index.js --epic "EPIC-42"

const { execSync, spawn } = require('child_process');
const fs = require('fs');
const path = require('path');
const readline = require('readline');

// ── State Management ──────────────────────────────────────────
const STATE_FILE = 'pipeline-state.json';

function loadState() {
  if (!fs.existsSync(STATE_FILE)) return createInitialState();
  return JSON.parse(fs.readFileSync(STATE_FILE, 'utf8'));
}

function saveState(state) {
  fs.writeFileSync(STATE_FILE, JSON.stringify(state, null, 2));
  log('STATE', `Phase: ${state.phase} | Status: ${state.status}`);
}

function createInitialState() {
  return {
    phase: 'INTAKE', status: 'RUNNING',
    epic: null, stories: [], architecture: null,
    sprint_plan: null, implementation: {}, test_results: null,
    deploy_manifest: null, human_queue: [], decisions_log: [],
    retry_counts: {}, created_at: new Date().toISOString()
  };
}

// ── Phase Definitions ─────────────────────────────────────────
const PHASES = [
  { id: 'INTAKE',       next: 'STORIES',        agent: 'intake'      },
  { id: 'STORIES',      next: 'STORY_GATE',     agent: 'story'       },
  { id: 'STORY_GATE',   next: 'ARCHITECTURE',   human: true          },
  { id: 'ARCHITECTURE', next: 'ARCH_GATE',      agent: 'architect'   },
  { id: 'ARCH_GATE',    next: 'PLANNING',       human: true          },
  { id: 'PLANNING',     next: 'IMPLEMENTATION', agent: 'planner'     },
  { id: 'IMPLEMENTATION',next: 'TESTING',        agent: 'dev'          },
  { id: 'TESTING',      next: 'DEPLOY_GATE',    agent: 'tester'      },
  { id: 'DEPLOY_GATE',  next: 'DEPLOYMENT',     human: true,  hard: true },
  { id: 'DEPLOYMENT',   next: 'COMPLETE',       agent: 'devops'      },
];

// ── Agent Runner ──────────────────────────────────────────────
async function runAgent(agentName, state) {
  log('AGENT', `Starting ${agentName} agent...`);

  const systemPrompt = fs.readFileSync(
    `./agent-prompts/${agentName}.md`, 'utf8'
  );

  const userMessage = `
Here is the current pipeline state:
\`\`\`json
${JSON.stringify(state, null, 2)}
\`\`\`

Execute your role for phase: ${state.phase}

CRITICAL RULES:
1. Output ONLY valid JSON matching the UpdateStateAction schema
2. If you need human input, set action="PAUSE" with a clear question
3. If you can proceed, set action="CONTINUE" with your work in updates
4. Never skip steps. Never assume. Always verify.
  `;

  // Call Claude via CLI (claude-code headless mode)
  const result = execSync(`echo '${JSON.stringify({systemPrompt, userMessage})}' | \
    claude --headless --output-format json \
    --max-turns 20`, { maxBuffer: 10 * 1024 * 1024 });

  return JSON.parse(result.toString());
}

// ── Human Gate Handler ────────────────────────────────────────
async function waitForHuman(question, state, isHard = false) {
  log('GATE', isHard ? '⛔ HARD GATE — Human approval required' : '⏸  Human input needed');

  // Show the question prominently
  console.log(`\n${'═'.repeat(60)}`);
  console.log(`🤖 AGENT QUESTION:\n`);
  console.log(question);
  console.log(`${'═'.repeat(60)}\n`);

  // Optional: send Slack notification
  if (process.env.SLACK_WEBHOOK) {
    await sendSlackNotification(question, state.phase);
  }

  // Wait for terminal input
  const rl = readline.createInterface({input: process.stdin, output: process.stdout});
  return new Promise(resolve => {
    rl.question('Your response (or type APPROVE): ', answer => {
      rl.close();
      resolve(answer);
    });
  });
}

// ── Main Pipeline Loop ────────────────────────────────────────
async function runPipeline(epicInput) {
  let state = loadState();
  state.epic = { raw: epicInput };
  saveState(state);

  log('START', `Pipeline starting. Epic: ${epicInput.substring(0,60)}...`);

  while (state.phase !== 'COMPLETE') {
    const phase = PHASES.find(p => p.id === state.phase);
    if (!phase) throw new Error(`Unknown phase: ${state.phase}`);

    if (phase.human) {
      // Human gate — wait for approval
      const question = state.human_queue[0] || `Phase ${state.phase} complete. Approve to continue to ${phase.next}?`;
      const answer = await waitForHuman(question, state, phase.hard);

      state.decisions_log.push({
        phase: state.phase, human_answer: answer,
        timestamp: new Date().toISOString()
      });
      state.human_queue = [];

      if (answer.toUpperCase().includes('APPROVE') ||
          answer.toUpperCase().includes('YES') ||
          answer.toUpperCase().includes('DEPLOY')) {
        state.phase = phase.next;
      } else {
        // Route feedback back to the previous agent
        state.human_feedback = answer;
        state.phase = PHASES[PHASES.findIndex(p => p.id === state.phase) - 1].id;
      }
    } else {
      // Agent phase — run autonomously
      const result = await runAgent(phase.agent, state);

      if (result.action === 'PAUSE') {
        state.human_queue.push(result.question);
        // Inject a gate before the next phase
        insertDynamicGate(state, result.question);
      } else if (result.action === 'CONTINUE') {
        // Merge agent updates into state
        Object.assign(state, result.updates);
        state.phase = phase.next;
      } else if (result.action === 'RETRY') {
        // Agent wants to retry (e.g., test failure)
        state.retry_counts[state.phase] = (state.retry_counts[state.phase] || 0) + 1;
        if (state.retry_counts[state.phase] >= 3) {
          state.human_queue.push(`Agent stuck after 3 retries in ${state.phase}: ${result.reason}`);
          insertDynamicGate(state, result.reason);
        }
      }
    }

    saveState(state);
    await sleep(1000); // brief pause between phases
  }

  log('DONE', '✅ Pipeline complete! Check pipeline-state.json for full output.');
}

function log(tag, msg) {
  const ts = new Date().toISOString().substring(11,19);
  console.log(`[${ts}] [${tag.padEnd(8)}] ${msg}`);
}
function sleep(ms) { return new Promise(r => setTimeout(r, ms)); }

// ── Entry Point ───────────────────────────────────────────────
const epicInput = process.argv[2] || process.env.EPIC_TEXT;
if (!epicInput) {
  console.error('Usage: node index.js "Your epic description here"');
  process.exit(1);
}
runPipeline(epicInput).catch(console.error);

Agent Reasoning Prompts
How each agent is instructed to think autonomously and decide when to stop vs continue
🔑
The Secret: The magic is in the system prompt. Each agent is given: (1) a clear role, (2) an explicit decision tree for "when to ask vs proceed", (3) a structured JSON output format, and (4) self-verification steps. This combination eliminates random behavior.
📄 File: agent-prompts/dev.md — Dev Agent (most complex)
You are a Senior Java Developer agent operating autonomously inside an SDLC pipeline.
You receive pipeline-state.json. You implement user stories one by one.

## YOUR REASONING PROCESS (follow exactly):

### Step 1 — ORIENT
Read pipeline-state.json. Answer these questions to yourself:
- Which stories are assigned to me (phase=IMPLEMENTATION)?
- Which stories are DONE vs TODO (check implementation.completed[])?
- What story should I implement next (lowest story_id with all deps done)?
- Does the codebase already have relevant code I should build on?

### Step 2 — IMPLEMENT
For each story:
a) Read the acceptance criteria carefully
b) Identify which files to create/modify (service, controller, repository, dto, test)
c) Write the implementation using the patterns in CLAUDE.md
d) Run: bash("mvn test -pl {module} -q")
e) If tests PASS → mark story DONE, move to next story
f) If tests FAIL → read error, fix, retry (up to 3 times)
g) If still failing after 3 tries → set action=RETRY with reason

### Step 3 — SELF-VERIFY before outputting CONTINUE
Ask yourself:
□ Does new code follow the patterns in ARCHITECTURE.md?
□ Are all acceptance criteria covered by the implementation?
□ Did mvn test pass with zero failures?
□ Are there any TODO comments left in the code?
□ Did I create/update unit tests for the new code?
If ANY answer is NO → fix it before outputting CONTINUE.

### Step 4 — DECIDE
- If ALL stories done AND tests pass → action=CONTINUE
- If a story fails 3 times → action=RETRY with specific error
- If you discover a missing requirement not in the stories → action=PAUSE
  with question: "Story US-XXX requires [missing thing]. Options: A, B, C. Which do you prefer?"
- NEVER ask about implementation details you can decide yourself
- ONLY ask about PRODUCT decisions (what should the feature do?)

## OUTPUT FORMAT (always valid JSON):
{
  "action": "CONTINUE | PAUSE | RETRY",
  "updates": {              // only on CONTINUE
    "implementation": {
      "completed": ["US-001", "US-002"],
      "test_results": { "passed": 47, "failed": 0 },
      "files_changed": ["user-service/src/.../UserService.java"]
    }
  },
  "question": "...",        // only on PAUSE — be specific, provide options
  "reason": "...",          // only on RETRY — include error message
  "reasoning_trace": "..."  // always — brief explanation of what you did
}
📄 File: agent-prompts/story.md — Story Agent
You are a Product Owner agent. You decompose epics into user stories.

## AUTONOMOUS DECISION RULES:
Proceed without asking if:
- Epic has clear user types, business goals, and scope
- Stories fit naturally into known services (user-service, order-service, frontend)
- Total story points are between 15–60 (reasonable for 3 sprints)

Stop and ask (action=PAUSE) if:
- Epic mentions an unknown external system (payment gateway, SMS, etc.)
  → Ask: "Epic mentions Stripe integration. Should I include payment stories in scope?"
- Total estimated points exceed 80 (too big for one pipeline run)
  → Ask: "This epic is 95 points. Recommend splitting into Phase 1 (auth+users, 45pt) and Phase 2 (orders, 50pt). Proceed with Phase 1?"
- Business rules are ambiguous
  → Ask: "Epic says 'managers can view orders'. Should managers see ALL orders or only their team's?"
  → NEVER ask: "Should I use GET or POST for the login endpoint?" — decide that yourself.

## STORY POINT CALIBRATION:
1pt: CSS/UI tweak, config change
2pt: Simple CRUD (1 entity, no business logic)
3pt: Standard feature (validation + service logic + tests)
5pt: Feature with business rules or multi-entity
8pt: Cross-service feature or complex UI
13pt: Needs architecture discussion → split first

Always justify points. Output stories sorted by dependency order.

Human Gate Design — What Gets Asked vs What Doesn't
The most important part of autonomous agents: getting the interruption balance right
✅ Agent CAN Decide Alone (Never Ask You)
CategoryExamples
Technical choicesGET vs POST, class names, package structure, which design pattern to use
Code styleVariable names, method ordering, import order
Test approachHow to mock, which assertions to use, test method names
Error handlingRetry logic, exception types, log levels
Minor estimatesAdjusting story from 3pt to 5pt based on discovery
DependenciesWhich Maven libraries to use for a given need
⏸ Agent MUST Pause and Ask (Product Decisions)
TriggerExample Question Agent Sends
Ambiguous business rule"When a MANAGER cancels an order, should it notify the CUSTOMER? Options: A) Yes via email, B) Yes via in-app, C) No notification"
Missing requirement"The login story doesn't specify password reset flow. Include it in this sprint or defer to Sprint 2?"
New external dependency"Order confirmation requires email. Should I integrate SendGrid now or use a stub and defer?"
Scope creep detected"Implementing orders requires an inventory check. This wasn't in the epic. Add a story or skip the check?"
Architecture conflict"Story US-012 requires order-service to call user-service. This creates coupling. Preferred: A) Allow REST call, B) Duplicate user data, C) Use events"
3 failed retries"The JWT refresh test failed 3 times. Error: [paste]. Need direction: A) Simplify to stateless JWT, B) Use Redis for token storage, C) Review my approach"
⛔ Hard Gates — Cannot Be Auto-Approved
  • Architecture decisions — service boundaries, data ownership, API contracts
  • Production deployments — any kubectl apply to a production cluster
  • Deleting data — DROP TABLE, bulk deletes, schema migrations with data loss
  • Security model changes — RBAC rules, authentication mechanisms
  • Cost-incurring operations — scaling up cluster nodes, enabling paid services
🤔 Agent Decision Tree: "Should I ask the human or proceed?"
Q Is this a PRODUCT decision (what the software should do) or IMPLEMENTATION decision (how to build it)?
PROD Always ask the human. Product decisions shape user experience and cannot be assumed.
IMPL Decide yourself using best practices. Log your decision in reasoning_trace.
Q Does proceeding risk data loss, cost, or security impact?
YES Always pause for human gate, regardless of everything else.
NO Proceed autonomously. Document your reasoning.

Monitoring & Audit Trail
How to see what your agents are doing and why
🗂 decisions-log format

# Terminal 1: run pipeline
node orchestrator/index.js "Epic..."

# Terminal 2: watch state file
watch -n 2 'cat pipeline-state.json | \
  python3 -m json.tool | head -30'

# Terminal 3: watch logs
tail -f decisions-log/latest.json | \
  grep -E "action|reasoning|phase"

# Terminal 4: watch test results

  mvn test -q 2>&1 | tail -5'

RETRY 3x same phaseReal problem — needs humanRead the error in decisions log, give specific direction
PAUSE with vague questionAgent prompt needs improvementAnswer it, then update the agent prompt to handle this case next time
Tests pass but coverage < 80%Test agent incompleteSay: "Focus on [ClassName] which has 0% coverage"
Smoke test fails post-deployEnv difference staging vs prodCheck env vars, secrets, DB connection strings
🔄 Resume a Paused Pipeline

If you need to stop and come back later, or if something crashes:

# Pipeline saves state after every phase
# To resume from where it left off:
node orchestrator/index.js --resume

# To resume from a specific phase:
node orchestrator/index.js --resume --from TESTING

# To replay a phase with different input:
node orchestrator/index.js --replay ARCHITECTURE \
  --feedback "Use event-driven between services instead of REST"
