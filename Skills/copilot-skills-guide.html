Skill 2 — Story Generator
Reads 6 docs → produces Jira-ready Type A + B stories with points and risk
📋 Type A — Reverse Stories
  • What: Documents existing behaviour as user stories
  • Source: SERVICE_PROFILE.md API surface table
  • Why: Creates regression baseline — "what must not change"
  • Sprint: Baseline (no dev work, just documentation)
  • Format: User story + acceptance criteria from actual code logic + auth model label
🔧 Type B — Gap Stories
  • P1 (Sprint 0): Security gaps, GKE failures, failing tests
  • P2 (Sprint 1-2): Test coverage, reliability, missing timeouts
  • P3 (Dedicated): Framework upgrades, major refactors
  • Source: APIGEE_AUTH_MAP findings + GKE_READINESS fails + TEST_HEALTH zero-coverage
  • Rule: P3 upgrades always in their own sprint — never bundled

User Story
As a customer application, I can authenticate and access user data
So that I can display profile information in my portal.

Implementation  UserController.getUserById → UserService.findById

- Then 200 returned with UserResponse DTO
- And 404 if user not found with {code:"USER_NOT_FOUND",message,timestamp}

Story Points: 3 | Risk: HIGH | Sprint: Baseline

Document Format — Token-Efficient Design
Why the docs are structured the way they are — and what to expect
🪙
Design principle: Each document is designed to answer a specific set of questions using the fewest tokens. Tables over prose. Symbols (✅⚠️❌) over words. Class names without package paths. Config key names only, never values. A full service's 6 docs should total under 800 lines — scannable in under 5 minutes by any developer.
📊 CLAUDE.md Structure (core training doc)
# CLAUDE.md — user-service



## CRITICAL RULES
1. NEVER modify customer-facing APIs without explicit human approval
2. NEVER add Basic Auth parsing — Apigee handles it upstream
3. NEVER hardcode secrets — use K8s Secrets / ConfigMap refs

## Auth Model
- Pattern A: Apigee validates → service sees X-Consumer-ID header only
- Pattern B: JWT forwarded → JwtAuthFilter re-validates locally
- Public: @PublicEndpoint → /actuator/**, /api/v1/health

| Tests | JUnit 4, Mockito, H2 | BaseServiceTest |

| JVM Xmx | 384m |

    Go to Claude.ai → Projects → Create New Project

    For each project: open "Custom Instructions"

    Add the skill .md files as Project knowledge

    Upload SKILL.md as a Knowledge document in each project — this ensures Claude can reference the skill's doc templates when generating outputs.

  4. 4

Token Efficiency — Why the Docs Are Designed This Way
8 rules baked into every document the archaeology skill produces
📉 Design Rules Applied to All Docs
  1. 1
    Tables over paragraphs — always

    A 10-row table communicates the same information as 10 paragraphs at ~20% of the token cost.

  2. 2
    ✅ ⚠️ ❌ over words

    "Pass / Warning / Fail" costs 3x more tokens than the symbol equivalents.

  3. 3
    Purpose cells: 3 words max

    "PostgreSQL user database" not "A PostgreSQL relational database that stores all user account information."

  4. 4
    Class names without package paths

    UserService not com.company.platform.user.service.UserService unless ambiguous.

📉 More Token Rules
  1. 5
    Config key names only — never values

    Write DB_PASSWORD not the actual password. Prevents secrets in docs AND saves tokens.

  2. 6
    Redact URLs to pattern

    {order-service-url}/api/v1/orders not http://order-service.production.svc.cluster.local:8082/api/v1/orders

  3. 7
    Omit implied Spring annotations

    @Service, @Component, @Repository are implied by class names — no need to list them.

  4. 8
    Repo-QA loads minimum docs per question

    Auth question → loads 2 docs (~1,500 tokens). Re-scanning source → 15,000–40,000 tokens. 10–25x saving.
