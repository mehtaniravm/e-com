Copilot Setup for Legacy Microservices

💡

🔧 Required Extensions & Tools
  1. 1
    GitHub Copilot + Copilot Chat (VS Code)

    Install GitHub.copilot + GitHub.copilot-chat. Requires GitHub Copilot Business or Enterprise license. Sign in with your GitHub org account.

  2. 2
    Open each microservice as its own VS Code workspace

    Don't open all services in one mega-workspace — Copilot context quality drops. One service = one workspace window. Use File → Add Folder to Workspace for shared libraries only.

  3. 3
    Install Kubernetes + Cloud Code extensions (for GKE context)

    ms-kubernetes-tools.vscode-kubernetes-tools + GoogleCloudTools.cloudcode. These let Copilot understand your K8s manifests and GKE deployment context.

  4. 4
    Create .github/copilot-instructions.md in each service repo

    Copilot reads this automatically. This is how you "train" Copilot on your service. See the template below.

📄 .github/copilot-instructions.md — Template Per Service

This file is the most important setup step. Copilot reads it for every suggestion in this repo.

# copilot-instructions.md
# Copilot context for: user-service

## SERVICE CONTEXT
- Service: user-service | Java 11 | Spring Boot 2.7.x | PostgreSQL
- Part of a production microservices platform on GKE (Google Kubernetes Engine)
- Customer APIs exposed via Apigee API Gateway

## AUTH MODEL (critical for any API work)
- External (customer-facing) APIs: protected by Apigee Basic Auth at gateway
  Apigee validates credentials before request reaches this service
  This service does NOT see raw Basic Auth headers — receives forwarded headers only
- Internal JWT flows: Apigee passes JWT in Authorization: Bearer {token}
  JwtAuthFilter validates token locally. Token claims: userId, roles[], sessionId, exp
- Public endpoints: annotated @PublicEndpoint (custom annotation)
- DO NOT add raw Basic Auth validation in service code — Apigee handles it

## GKE DEPLOYMENT
- Namespace: production | Staging: staging
- Config via ConfigMaps + Secrets (never hardcode values)
- Health: /actuator/health (liveness) | /actuator/health/readiness (readiness)
- Resources: requests.memory=256Mi, limits.memory=512Mi, cpu limits=500m
- HPA enabled: min=2, max=8 replicas on CPU 70%

## CODING PATTERNS (follow exactly, don't invent new ones)
- Exception: GlobalExceptionHandler → error format {code, message, timestamp}
- DTOs: separate Request/Response. Use MapStruct. Never expose @Entity.
- Logging: SLF4J + MDC(requestId, userId). No System.out.println ever.
- Testing: JUnit 4 (NOT 5). Mockito. H2 in-memory. Base class: BaseServiceTest.
- Transactions: @Transactional on service layer only, never controllers.

## CUSTOMER API CONTRACTS
- /api/v1/users/** : CustomerA — DO NOT change response structure
- /api/v1/auth/**  : CustomerB — DO NOT change without versioning

## NEVER DO
- Never hardcode secrets, passwords, or API keys
- Never add System.exit() or Runtime.halt()
- Never bypass JwtAuthFilter for non-@PublicEndpoint routes
- Never use RestTemplate for new code (use WebClient instead)

Apigee + Auth — Understanding Your Gateway Layer
Critical context for Copilot to understand before touching any security code
🗺 Your Auth Architecture — What Copilot Must Know
CUSTOMER REQUEST FLOW:
═══════════════════════════════════════════════════════════════

  Customer App  ──────────→  Apigee API Gateway  ──────────→  GKE Service
                              (Google Cloud)                   (Internal)

PATTERN A — Basic Auth (most external APIs):
  Customer:  GET /api/v1/orders  +  Authorization: Basic base64(user:pass)
  Apigee:    Validates credentials against Apigee app credentials
             Strips Basic Auth header
             Adds: X-Consumer-ID, X-API-Key, X-Apigee-App-Name headers
  Service:   Receives request with NO Authorization header
             Trusts X-Consumer-ID for identity (set by Apigee)
             No auth validation needed in service code ← KEY POINT

PATTERN B — JWT Passthrough (admin/partner APIs):
  Customer:  POST /api/v1/admin/users  +  Authorization: Bearer {JWT}
  Apigee:    Validates JWT signature (public key)
             Forwards full request INCLUDING Authorization: Bearer header
  Service:   JwtAuthFilter intercepts → re-validates JWT locally
             Extracts: userId, roles[], sessionId from claims
             Sets MDC context with these values
             Route proceeds if valid, 401 if not

WHAT THIS MEANS FOR COPILOT:
  - Never generate Basic Auth parsing code in service layer
  - JwtAuthFilter is the ONLY JWT validation point in service
  - X-Consumer-ID header is the authoritative customer identity for Pattern A
  - /actuator/** must ALWAYS be excluded from JwtAuthFilter
  - Any new endpoint must be labeled: BASIC_AUTH_VIA_APIGEE | JWT | PUBLIC

Prompts and config so Copilot generates GKE-correct code and manifests
📄 Add to copilot-instructions.md — GKE Section
# GKE Deployment Context

## Cluster Setup
- Project: [your-gcp-project-id]
- Cluster: production-cluster (us-central1) | staging-cluster (us-central1)
- Namespaces: production | staging | monitoring
- Ingress: GKE Nginx Ingress Controller (NOT GKE native ingress)
- Service mesh: None (plain HTTP between services in cluster)
- DNS: service-name.namespace.svc.cluster.local

## ConfigMap and Secret Conventions
# DB URL comes from ConfigMap:
spring.datasource.url=${DB_URL}   # → K8s ConfigMap: db-config
spring.datasource.password=${DB_PASSWORD}  # → K8s Secret: db-secret

# JWT Secret:
jwt.secret=${JWT_SECRET}   # → K8s Secret: jwt-secret

## Deployment Conventions
- image: gcr.io/[project]/[service-name]:${IMAGE_TAG}
- imagePullPolicy: Always
- replicas: 2 minimum (never 1 in production)
- rollingUpdate: maxSurge=1, maxUnavailable=0

## JVM Memory Rule
Container memory limit: 512Mi → JVM: -Xmx384m -Xms256m
Container memory limit: 1Gi  → JVM: -Xmx768m -Xms512m
(Always set both -Xmx and -Xms to avoid heap resize thrashing)

## Health Check Endpoints
livenessProbe:  GET /actuator/health/liveness   (is the app alive?)
readinessProbe: GET /actuator/health/readiness  (is the app ready for traffic?)
startupProbe:   GET /actuator/health            (initial startup, 30s window)
