The Core Concept — Same Brain, Copilot Body
The reasoning loop is identical. Only the runtime changes.
💡
Key Insight: The Queen pattern doesn't care which AI runs the agents. The Queen is a state machine + reasoning loop. In the Claude edition, agents ran via claude -p. In this edition, the same agents run via copilot -p (Copilot CLI) or as GitHub Coding Agent tasks. The pipeline-state.json, the gates, and the agent prompts carry over almost unchanged.
❌ Copilot As Most People Use It (Manual)
  • You open Copilot Chat, type a prompt, read the output
  • You copy output and paste it into the next prompt
  • You manually check: "did the tests pass?"
  • You decide when to move from stories → architecture
  • If you forget a step, it's skipped
  • Each chat session has no memory of the last
✅ Copilot As Orchestra (Autonomous)
  • You drop in a Jira Epic URL → pipeline starts
  • Queen runs each phase via Copilot CLI or Coding Agent
  • Each agent reads the previous agent's output from state
  • Tests run, fail → Dev agent retries and fixes (3x max)
  • Pipeline pauses only at real decision points (gates)
  • Full audit log of every decision in pipeline-state.json
🔄 The Reasoning Loop — Identical to the Claude Edition
OBSERVE  → Read current state (pipeline-state.json, test results, Jira, previous output)
ORIENT   → What phase am I in? What's my goal? What's blocking me?
DECIDE   → Can I proceed autonomously OR do I need human input?
ACT      → Execute tools (write code, run tests, update Jira, open PR)
RECORD   → Write decision + rationale to pipeline-state.json
REPEAT   → Loop back to OBSERVE unless phase is COMPLETE or BLOCKED

This is the ReAct pattern (Reasoning + Acting). The Copilot CLI supports it natively: it plans, calls tools, observes results, and iterates — the same agentic loop Claude Code runs.

🔀
Two Orchestration Models — Pick One (or Mix)
Copilot gives you a choice the Claude edition didn't have
Model A — Local Loop (Copilot CLI)

A Node.js Queen script on your machine (or a runner VM) calls copilot -p for each phase — the direct equivalent of the Claude edition's claude -p loop.

  • Closest to the Claude edition — same queen.js, swap the CLI call
  • Full control over every phase, tool, and retry
  • Gates = console prompts or Slack approvals
  • Runs on your laptop or a CI runner
  • Best for: staging pipelines, experimentation, full visibility
Model B — Cloud Loop (GitHub Coding Agent)

The Queen creates GitHub Issues per story and assigns them to Copilot (the Coding Agent). Copilot works in GitHub Actions, opens PRs. Your PR review IS the gate.

  • Zero infrastructure — GitHub runs the agents for you
  • Native gates: branch protection + required PR reviews
  • Agent works in an isolated Actions environment
  • Audit trail = Issues, PR history, session logs
  • Best for: production-grade flow, team-wide rollout
🏆
Recommended hybrid: Use Model A for the planning phases (Story → Architecture → Sprint Plan) where outputs are documents, and Model B for implementation (Dev → Test) where outputs are PRs and GitHub's review gates shine. The Queen script bridges both: it writes plans locally, then creates Issues and assigns them to Copilot.

The Autonomous Pipeline — Phase by Phase
Exactly what runs automatically vs what needs you — Copilot runtime per phase
GATE 📥 Phase 0 — Epic Intake ~2 min · Model A
You provide: Jira Epic URL or pasted Epic text — the only manual trigger in the whole pipeline.
Queen does: node queen.js --epic https://jira.../E-42 → fetches Epic via Jira MCP, writes it into pipeline-state.json, sets phase to STORY_GENERATION.
AUTO 📝 Phase 1 — Story Generation ~5-10 min · Model A · copilot -p
Queen runs: copilot -p "$(cat .github/prompts/story-agent.prompt.md)" --allow-tool read --allow-tool write. The Story Agent reads the Epic + .claude/context/apis.md + auth.md, generates Type A/B stories with Fibonacci points, acceptance criteria, auth labels, and writes them into pipeline-state.json. Retry logic: if output fails JSON schema validation, Queen re-prompts with the error (max 3x).
GATE 1 🙋 Phase 2 — Story Approval ~10 min of your time
Queen prints the story table to console + posts to Slack: "14 stories, 52 points. Approve to proceed? [y/n/edit]". You can edit stories inline (Queen re-opens your $EDITOR with the story JSON). On approval, Queen pushes stories to Jira via MCP and advances to ARCHITECTURE.
AUTO 🏛 Phase 3 — Architecture ~10 min · Model A · copilot -p
Architect Agent prompt runs with approved stories + deps.md + gke.md context. Produces ARCHITECTURE.md: service boundaries, API contracts, data model changes, Apigee impact, GKE resource changes. Queen validates the output contains all required sections before advancing.
PAUSE 🧐 Phase 4 — Architecture Review ~5 min of your time
Soft pause: Queen shows the architecture summary and waits 30 min for objections (configurable). No response = auto-proceed. Objection = Queen re-runs Architect Agent with your feedback appended. This is a PAUSE, not a GATE — designed for speed.
AUTO 📅 Phase 5 — Sprint Planning + Issue Creation ~5 min · Model A → hands off to Model B
Planner Agent orders stories by dependency, splits parallel tracks. Then the Queen bridges to Model B: for each story it runs gh issue create --title "US-G001: ..." --body-file story-US-G001.md --label copilot-pipeline and assigns each issue to @copilot — the GitHub Coding Agent. Each issue body contains the story, acceptance criteria, links to ARCHITECTURE.md, and the repo's context files do the rest (Copilot Coding Agent reads .github/copilot-instructions.md + AGENTS.md automatically).
AUTO 💻 Phase 6 — Implementation (Coding Agent) ~30-90 min per story · Model B · cloud
The GitHub Copilot Coding Agent picks up each assigned issue, spins up an isolated GitHub Actions environment, reads AGENTS.md + copilot-instructions.md + CodeFixes/, implements the story, runs mvn test in its environment, and opens a draft PR per story. Failing tests → the agent iterates in-session. You can watch progress live in the PR's session log.
AUTO 🧪 Phase 7 — Test & CI Verification ~10 min per PR · GitHub Actions
Your existing CI workflow runs on each Copilot PR: unit tests, JaCoCo coverage threshold (80%), Trivy scan, customer contract tests (Playwright). Queen polls PR check status via gh pr checks. Red checks → Queen comments on the PR with the failure log and @copilot mention — the Coding Agent picks the comment up and pushes fixes (max 3 rounds, then escalates to human_queue).
GATE 2 ✅ Phase 8 — PR Review = The Native Gate ~15 min of your time
This is where Copilot's model beats a custom gate: branch protection requires your review on every Copilot PR. You review the diff like any teammate's PR. Request changes → Coding Agent revises. Approve + merge → Queen detects the merge and advances. No custom approval code needed — GitHub's review system IS the gate.
GATE 3 🚀 Phase 9 — Deploy to GKE ~10 min · GitHub Actions + Environments
Merge to main auto-deploys to staging (GKE) + smoke tests. Production deploy uses a GitHub Environment with required reviewers — the deploy job pauses until you click Approve in the Actions UI. Post-deploy: smoke tests + customer contract verification. Failure → automatic kubectl rollout undo + alert.

The Queen Orchestrator — queen.js for Copilot
Same skeleton as the Claude edition — the agent runner is swapped
👑 queen.js — Core Loop (Copilot backend)
// queen.js — Queen Orchestrator, GitHub Copilot edition
// Prereqs: npm i -g @github/copilot   |   gh auth login
const { execSync, spawnSync } = require("child_process");
const fs = require("fs");

const STATE = "pipeline-state.json";
const state = () => JSON.parse(fs.readFileSync(STATE));
const save  = (s) => fs.writeFileSync(STATE, JSON.stringify(s, null, 2));

// ── Agent runner: Model A (local Copilot CLI) ──────────────
function runCopilotAgent(promptFile, extraContext = "") {
  const prompt = fs.readFileSync(promptFile, "utf8")
    + "\n\n## CURRENT PIPELINE STATE\n"
    + fs.readFileSync(STATE, "utf8") + extraContext;

  // Non-interactive run; tools whitelisted per-phase; logs kept
  const r = spawnSync("copilot", [
    "-p", prompt,
    "--allow-tool", "read",
    "--allow-tool", "write",
    "--allow-tool", "shell(mvn test)",   // scoped shell access
    "--log-dir", "logs/",
    "--log-level", "debug"
  ], { encoding: "utf8", maxBuffer: 64 * 1024 * 1024 });

  if (r.status !== 0) throw new Error("Agent failed: " + r.stderr);
  return r.stdout;
}

// ── Agent runner: Model B (GitHub Coding Agent via Issues) ──
function assignStoryToCodingAgent(story) {
  fs.writeFileSync(`story-${story.id}.md`, renderStoryBody(story));
  const url = execSync(
    `gh issue create --title "${story.id}: ${story.title}" ` +
    `--body-file story-${story.id}.md --label copilot-pipeline`,
    { encoding: "utf8" }).trim();
  // Assign to Copilot Coding Agent — it picks the issue up automatically
  execSync(`gh issue edit ${url} --add-assignee copilot`);
  return url;
}

// ── Gate helper (console; swap for Slack later) ────────────
function gate(message) {
  console.log("\n🛑 GATE: " + message);
  const answer = require("readline-sync").question("[y/n/edit] > ");
  if (answer === "edit") execSync(`${process.env.EDITOR} ${STATE}`, {stdio:"inherit"});
  return answer !== "n";
}

// ── Main state machine ──────────────────────────────────────
const PHASES = {
  STORY_GENERATION: () => {
    runCopilotAgent(".github/prompts/story-agent.prompt.md");
    return "STORY_APPROVAL";
  },
  STORY_APPROVAL: () => {
    if (!gate(`${state().stories.length} stories generated. Approve?`))
      return "STORY_GENERATION";       // regenerate with feedback
    execSync("node push-stories-to-jira.js");
    return "ARCHITECTURE";
  },
  ARCHITECTURE: () => {
    runCopilotAgent(".github/prompts/architect-agent.prompt.md");
    return "SPRINT_PLAN";
  },
  SPRINT_PLAN: () => {
    runCopilotAgent(".github/prompts/planner-agent.prompt.md");
    // Bridge to Model B — create issues, assign to Coding Agent
    const s = state();
    s.issues_created = s.sprint_plan.ordered_stories
      .map(story => assignStoryToCodingAgent(story));
    save(s);
    return "AWAIT_PRS";
  },
  AWAIT_PRS: () => {
    // Poll PR checks; comment @copilot on failures (max 3 rounds)
    execSync("node poll-and-heal-prs.js", {stdio:"inherit"});
    return "DONE";  // merges + deploy are gated natively in GitHub
  }
};

let phase = state().phase || "STORY_GENERATION";
while (phase !== "DONE") {
  console.log(`\n▶ PHASE: ${phase}`);
  phase = PHASES[phase]();
  const s = state(); s.phase = phase; save(s);
}
🔁 poll-and-heal-prs.js — Self-Healing CI
// For each open copilot-pipeline PR:
//  1. gh pr checks <n> --json
//  2. if failing and rounds < 3:
//     gh pr comment <n> --body
//     "@copilot tests failing:
//      <paste failure log>
//      Fix the test or the code,
//      do not change the contract."
//  3. Coding Agent sees the mention,
//     pushes a fix to the same PR
//  4. rounds == 3 → human_queue +
//     Slack alert, stop touching PR
🧯 Retry & Escalation Rules
  • Schema-invalid agent output → re-prompt with validation error, max 3x
  • Failing PR checks → @copilot comment with logs, max 3 rounds
  • Same error twice in a row → escalate immediately (don't loop)
  • Customer contract test failure → always escalate, never auto-fix
  • Anything in human_queue → pipeline pauses that story, others continue

---
mode: agent
description: Generate Type A + B stories from an Epic using repo context
---
You are the STORY AGENT in an autonomous SDLC pipeline.

## Inputs
- The Epic and current pipeline state are appended below this prompt.
- Read .claude/context/apis.md and .claude/context/auth.md for service context.

## Your Job
1. Generate user stories: Type A (existing behaviour) + Type B (gaps/new work)
2. Each story: title, user story, acceptance criteria (max 5, Given/When/Then),
   Fibonacci points (1/2/3/5/8 — 13 means split), auth label
   (BASIC_AUTH_VIA_APIGEE | JWT | PUBLIC), risk (HIGH/MED/LOW)
3. Write the stories array into pipeline-state.json under "stories"
4. Do NOT invent endpoints — only reference what exists in apis.md

## Output Contract (validated by the Queen)
pipeline-state.json must parse, stories[] non-empty,
every story has: id, title, points, auth, risk, acceptance[]

The same file works three ways: the Queen pipes it through copilot -p, developers run it interactively as /story-agent in VS Code Copilot Chat, and the front-matter makes it an agent-mode prompt.

💡
AGENTS.md is the Coding Agent's system prompt. When a story issue is assigned to @copilot, the Coding Agent automatically reads AGENTS.md and .github/copilot-instructions.md from the repo. Your hard rules (never parse Basic Auth, never hardcode secrets, JVM = 75% of container limit) apply to every story with zero extra prompting — exactly like CLAUDE.md does for Claude Code.

Human Gates — Native GitHub Mechanisms
The Copilot edition's biggest advantage: two of three gates need zero custom code
🚦 The Three Gates
GateWhat You ApproveMechanismCustom Code?
GATE 1 — Stories Story list, points, scope before Jira push Queen console prompt (or Slack interactive message) ~10 lines in queen.js
GATE 2 — Code Every PR the Coding Agent opens Branch protection + required PR review — Copilot's PRs cannot merge without your approval; "Request changes" sends the agent back to work None — native GitHub
GATE 3 — Production Deploy Staging → Production promotion GitHub Environment "production" with required reviewers — deploy job pauses in Actions UI until you approve None — native GitHub

✓ Require approvals: 1
✓ Dismiss stale approvals on new commits
✓ Require status checks: ci / tests,
  ci / coverage, ci / contract-tests
✓ Do not allow bypassing the above

# Result: Coding Agent PRs are

# Settings → Environments → production:
✓ Required reviewers: you (+ lead)
✓ Wait timer: 0 min
✓ Deployment branches: main only

# deploy.yml:

  deploy-prod:
    environment: production   # ← pauses
    runs-on: ubuntu-latest    #   here for

Decision rule for the agents (same as Claude edition): proceed autonomously when the action is reversible and inside the approved plan; PAUSE when ambiguous; GATE when irreversible (Jira mass-push, customer-contract change, production deploy). Customer contract test failures always escalate to a human — never auto-fixed.

Setup — Zero to Running Pipeline
One-time, ~45 minutes
  1. 1
    Install the Copilot CLI + GitHub CLI 
    npm install -g @github/copilot
copilot          # sign in once interactively
gh auth login    # GitHub CLI for issues/PRs

    Requires Copilot Pro/Business/Enterprise. Coding Agent requires it enabled in org policy.

  2. 2
    Run Code Archaeology on the repo first

    The pipeline depends on the context files: AGENTS.md, copilot-instructions.md, .claude/context/*, CodeFixes/. The Coding Agent and CLI both read these automatically — they are the agents' institutional memory.

  3. 3
    Add the agent prompt files

    Create .github/prompts/story-agent.prompt.md, architect-agent.prompt.md, planner-agent.prompt.md, reviewer-agent.prompt.md (templates in this doc).

  4. 4
    Configure MCP for Jira 
    # .vscode/mcp.json (CLI + VS Code) — add your Jira MCP server
{ "servers": { "jira": { "url": "https://your-jira-mcp...", "type": "http" } } }

    The Coding Agent gets its own MCP config in repo settings → Copilot → Coding agent → MCP.

  5. 5
    Turn on the native gates

    Branch protection on main (GATE 2) + production environment with required reviewers (GATE 3). Five minutes in repo settings.

  6. 6
    Drop in queen.js + start 
    npm i readline-sync
node queen.js --epic https://yourjira/browse/E-42

    Watch Phase 1 generate stories, approve at GATE 1, then watch issues appear and the Coding Agent open PRs.
