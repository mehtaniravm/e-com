Role Update Propagation — SCRUM-11

⚠️
Stale JWT Window After Role Change
JWT has 15-min TTL. A role removed via admin won't take effect until the current AT expires. During that window the user retains elevated access.
Mitigation: Short AT TTL (15 min) + JWT blocklist (jti in Redis) for immediate revocation on sensitive role downgrades. On role update, signal auth-service to invalidate all tokens for userId.
Inter-service call: When PATCH /admin/users/{id}/roles is called, identity-service POSTs to auth-service/internal/revoke-user with the userId. Auth-service adds all active JTIs for that user to a Redis blocklist (TTL = max AT TTL = 15 min).

Inter-Service Communication Patterns

Pattern selection: Synchronous REST for operations requiring immediate consistency (stock reservation during order creation). Asynchronous events for decoupled state propagation (role changes, account deactivation) using a lightweight approach. All inter-service calls use mTLS within the cluster.
CallerCalleeOperationPatternFailure Mode
order-svc product-svc Reserve / Confirm / Release stock Sync REST Rollback reservation, return 422 to client
order-svc product-svc Fetch canonical prices Sync REST Circuit breaker → reject order
identity-svc auth-svc Revoke all tokens for userId (role change, deactivation) Sync REST (internal) Log + async retry; accept eventual consistency
All services auth-svc JWKS key fetch for JWT validation Sync REST + local cache Cached keys (TTL=1hr) tolerate auth-svc outage

Risks, Anti-Patterns & Mitigations

🔴
ANTI-PATTERN: Distributed Monolith (SCRUM-19)
If order-service calls product-service synchronously AND both share a DB, you've created a distributed monolith. The sync REST call for stock reservation creates runtime coupling — if product-service is down, orders cannot be placed even if it's a temporary outage.
FIX: Circuit breaker (Resilience4j) + bulkhead pattern. Consider event-driven saga for order creation flow at scale. Reserve stock with compensating transaction on failure.
🔴
RISK: JWT Stale Roles Window (SCRUM-11)
Access tokens are self-contained. A role downgrade (ADMIN → CUSTOMER) won't take effect for up to 15 minutes if the AT is still valid. This is a real privilege escalation window.
FIX: Redis-backed JTI blocklist + force-revocation endpoint on auth-service called synchronously by identity-service on role change. Keep AT TTL to 15 min max.
🔴
RISK: No Idempotency Key Enforcement (SCRUM-18)
Network retries on POST /orders without idempotency protection will create duplicate orders. Mobile clients and unreliable networks make this a production incident waiting to happen.
FIX: Require Idempotency-Key header (UUID v4/v7). Store key+response in Redis (TTL=24hr). Return cached response if key already seen. Return 409 if key processed with different payload.
🟡
ANTI-PATTERN: Email Stored in Both auth_db and identity_db
Email is the login identifier (auth_db) AND a profile field (identity_db). An email change in identity-service won't propagate to auth-service — login will stop working or use stale email.
FIX: Email changes must go through auth-service. Identity-service caches email as read-only display value. Or: use userId as the only cross-service identifier, email changes are auth-service only.
🟡
RISK: No Rate Limiting on Auth Endpoints (SCRUM-6,7)
POST /auth/login and /auth/register without rate limiting are open to brute-force and registration spam. A bot can attempt unlimited credential guesses.
FIX: Bucket4j rate limiter per IP (e.g. 10 login attempts / 15 min). Account lockout after 5 failed attempts (LOCKED status in credentials table). reCAPTCHA on register endpoint.
🟡
RISK: Product Price Staleness in Order Snapshots (SCRUM-20)
Order-service fetches live price from product-service at order creation time. If product-service is slow, this creates latency. The snapshot is correct at creation but there's no audit trail of what price was used.
FIX: Always snapshot unit_price in order_items (already in model). Add priceSnapshotAt timestamp. Consider caching prices in order-service (TTL=5min) to reduce sync call latency.
🟢
OBSERVATION: Shared Library for JWT Validation (SCRUM-9)
Extracting JWT validation into a shared library (JAR) creates a compile-time dependency. All services must upgrade together when the library changes. This is a controlled coupling tradeoff.
ACCEPTED: For a team this size, shared-lib is pragmatic. Use semantic versioning + compatibility matrix. Alternative at scale: validate JWT locally in each service via JWKS — no shared lib needed, truly decoupled.
🟢
OBSERVATION: No API Gateway in Current Design
GKE Ingress + Cloud Armor handles routing and DDoS, but there's no dedicated API gateway for cross-cutting concerns (rate limiting, request transformation, API versioning, analytics).
FUTURE: Consider Cloud Endpoints or Kong on GKE as the codebase grows. For current scope, Spring Cloud Gateway as a lightweight option adds rate limiting, auth delegation, and routing in one deployment.
