The Fundamental Difference
New app vs legacy retrofit — completely different starting point
🌱 Greenfield (Previous Guide)
  • Start from Epic → generate everything
  • Claude invents architecture fresh
  • No existing code conflicts
  • Clean test baseline (zero tests)
  • No production risk
🏚 Legacy Retrofit (This Guide)
  • Start from CODE → derive stories
  • Claude must learn YOUR architecture first
  • Existing patterns must be respected
  • Legacy tests may be broken, absent, or wrong
  • Every change risks breaking production
  • Customer integrations = extra risk surface
⚠️
Core Rule for Legacy: Claude must READ before it WRITES. The entire prework phase creates a comprehensive knowledge base so agents understand your codebase's patterns, antipatterns, dependencies, and business logic before generating a single story or touching a single file.
🗺 How Legacy SDLC Flow Differs
GREENFIELD:  Epic → Stories → Architecture → Code → Tests → Deploy

LEGACY:      Code Scan → Knowledge Base → Stories ← Code
                  ↓
             Gap Analysis (what's missing / broken)
                  ↓
             Legacy Fix Plan → Implement Fixes → Fix Tests
                  ↓
             New Feature Stories → Implement → New Tests
                  ↓
             E2E Playwright (UI + API + Customer API)
                  ↓
             Deploy (with rollback plan — always)

📄 File 3: DOMAIN_GLOSSARY.md — Teach Claude Your Business Language
## DOMAIN GLOSSARY (use these exact terms in stories and code)

User:      registered person in system. Has roles.
Customer:  EXTERNAL COMPANY that integrates with our APIs (not end-user)
Merchant:  type of User who can create orders
Order:     a transaction REQUEST — NOT the same as Purchase
Purchase:  a completed, paid Order. Different lifecycle.
SKU:       unit of inventory. Format: [CATEGORY]-[ID]-[VARIANT]

## BUSINESS RULES AGENTS MUST KNOW
- Orders cannot be cancelled after status=SHIPPED
- User deactivation requires 30-day notice if user has active Orders
- Inventory reservation happens at Order CONFIRMED, not CREATED
- Customer API calls must complete in <200ms (contractual SLA)
- All PII fields (email, phone, address) are encrypted at rest

## WHAT THIS SYSTEM IS NOT
- Not a payment processor (no financial transactions in code)
- Not a customer-facing storefront (B2B only)
- Not real-time inventory (15-min sync lag is acceptable)

Phase 2 — Story Generation FROM Code
Reverse-engineering what code does into stories + gap stories for what it should do
🔄 Two Types of Stories for Legacy Systems
Type A — Reverse Stories

Document what code CURRENTLY does. Become source of truth for preserved behavior. Used as regression protection baseline.

derived from codeno new dev needed
Type B — Gap Stories

What system SHOULD do but doesn't. Missing tests, broken edge cases, absent features, legacy code needing modernization.

requires dev worksprint-planned
🤖 Story Agent Prompt — Legacy Mode
You are a Product Owner analyzing an EXISTING production service.

Read these files first (in order):
1. CLAUDE.md (root) — critical rules and patterns
2. {service}/CLAUDE.md — service-specific quirks
3. {service}/SERVICE_PROFILE.md — what was found in code scan
4. {service}/TEST_HEALTH.md — current test state
5. DOMAIN_GLOSSARY.md — business terms

Then produce TWO story sets:

## PART A: REVERSE STORIES (existing behavior)
For each controller endpoint / major service method:
  Title: [Existing] {feature name}
  User Story: As a {actor}, I can {action} so that {business purpose}
  Current Implementation: {class + method}
  Acceptance Criteria: Given/When/Then matching EXISTING code behavior
  Story Points: [based on complexity of existing code]
  Risk Level: HIGH / MED / LOW

## PART B: GAP STORIES (missing/broken)
Look for:
- Endpoints with no tests → "Add test coverage for X"
- TODO/FIXME clusters → story per related cluster
- Missing validation → "Add input validation for X"
- Broken tests in TEST_HEALTH.md → "Fix failing tests for X"
- Deprecated deps with CVEs → "Upgrade {lib} to {version}"

Gap story format:
  Title: [Fix] or [Add] or [Upgrade] {specific thing}
  Story Points: effort to fix (not original effort)
  Priority: P1 (blocking/security) / P2 (quality) / P3 (tech debt)
  Risk: what breaks if this goes wrong?

## OUTPUT
Sort by service → risk (HIGH first)
Flag dependencies between stories
Total story points per service
Highlight P1 gaps that need immediate sprint planning

Phase 3 — Legacy Code Fixes
Safe approach to correcting legacy code without breaking production
🚨
Production Safety Rule: Every legacy fix MUST follow this order — (1) Write a test capturing current behavior FIRST, (2) verify test passes against current code, (3) make the fix, (4) verify same test still passes. Never fix without a test anchor. The test is your safety net.
🤖 Dev Agent Prompt — Legacy Fix Mode
You are modifying a PRODUCTION service running for years.
Every change is potentially breaking something a customer depends on.

## RULES FOR LEGACY CODE:
1. READ service CLAUDE.md first. Know the danger zones.
2. Before fixing ANYTHING, write a test capturing CURRENT behavior.
   Run it. It must pass. This is your safety anchor.
3. Make the MINIMAL change to fix the issue. No refactoring while fixing.
4. After fix, run ALL existing tests. Zero new failures allowed.
5. If a test was @Ignored with no explanation, do NOT re-enable it.
   Add comment: // RE-ENABLE PENDING: investigate why disabled
6. If you find new issues while fixing, do NOT fix them now.
   Add // FOUND: {description} comment. Create a gap story.

## APPROACH BY FIX TYPE:

### N+1 Queries
- Add @EntityGraph or JOIN FETCH
- Write before/after query count assertion
- Check if result order changes

### Missing Input Validation
- Use existing validation style (check what other fields use)
- Add to DTO Request object, NOT controller
- Return existing error format (check GlobalExceptionHandler)

### Upgrading Dependencies
- One dependency at a time
- Run full test suite after each upgrade
- Spring Boot major version = dedicated sprint + QA sign-off

Phase 4 — Test Modernization & Coverage
Fix existing tests, fill gaps, migrate JUnit 4 → 5 where safe
🤖 Test Agent Prompt — Legacy Mode
You are a Test Engineer on a legacy production codebase.
Improve test coverage without breaking existing tests.

## RULES FOR LEGACY TESTS:
1. Check JUnit version (pom.xml). JUnit 4 and 5 CANNOT mix without config.
   Use the SAME version as existing tests in that service.
2. Use the same base test class if one exists (BaseServiceTest, BaseControllerTest)
3. Follow naming pattern: {Class}Test.java in same package as production class
4. H2 or TestContainers — use whatever that service already uses

## TEST WRITING ORDER:
P1: Tests for code with ZERO coverage that customer APIs call
P2: Tests for business logic in Service classes
P3: Tests for every error/exception path (often 0% covered)
P4: Tests for edge cases mentioned in FIXME/TODO comments

## FOR EACH NEW TEST FILE:
1. Read the production class FULLY before writing one line of test
2. List all methods → what each does
3. List scenarios: happy path + each error path + each edge case
4. Write tests: happy paths first, then errors, then edge cases
5. Run: mvn test -pl {service} -Dtest={ClassName}Test -q
6. Report: pass/fail + coverage delta

## DO NOT:
- Do NOT test Spring Boot framework behavior (that's tested upstream)
- Do NOT write tests that only call getters/setters
- Do NOT use Thread.sleep() — use Awaitility if async
- Do NOT write tests requiring real network access

Phase 5 — Playwright E2E: UI + API + Customer API
Three test layers required for a system with customer integrations
🗂 Three Playwright Test Layers
Layer 1 — UI Tests

Browser-based tests against React/Angular UI. User journeys: login, navigate, create orders, manage users.

@uiPage Object pattern
Layer 2 — Internal API

HTTP tests against REST APIs directly. All endpoints, all verbs, auth flows, error codes. Fast, no browser.

@apirequest() fixture
Layer 3 — Customer Contracts

Simulate exactly what customer applications do. Their tokens, their endpoints. Most critical layer.

@contractgates every deploy

Phase 6 — Deployment for Legacy Systems
More caution than greenfield — rollback plan is non-optional
📋 Pre-Deploy Checklist (Agent-Verified)
Before any deployment, verify ALL of these:

## REGRESSION GATE
□ All existing tests pass (no NEW failures vs baseline)
□ Coverage % did not DROP below previous measurement
□ @smoke Playwright suite passes against staging
□ @contract customer API tests pass against staging

## CHANGE IMPACT GATE
□ DEPENDENCY_MAP.md checked — downstream services notified?
□ DB schema change? → migration script + rollback script present
□ API contract change? → new version path, old path preserved
□ Shared library version change? → ALL services retested

## CUSTOMER COMMUNICATION GATE
□ Breaking change? → customer notified N days ago per SLA
□ Deprecation notice sent + customer test env updated?

## ROLLBACK PLAN
□ Previous Docker image tag saved
□ kubectl rollout undo command verified in staging
□ DB rollback script tested in staging
□ Post-rollback smoke test ready

## MONITORING
□ Error rate baseline recorded before deploy
□ Alert thresholds set for 15-min post-deploy window

Do NOT proceed if ANY checkbox unchecked. Escalate to human.
🔄 Safe Deploy Strategy
  1. 1
    Deploy to staging first — run all 4 test suites

    Never skip staging. Customer contract tests here save production incidents.

  2. 2
    Canary deploy — 5% traffic to new version

    Watch error rates for 15 min before full rollout. Automated rollback if error rate spikes.

  3. 3
    Blue-green swap (safest for legacy)

    Keep old version running until new is verified. Instant rollback = switch traffic back.

  4. 4
    Post-deploy: run @smoke + @contract against production

    If any fail → automated rollback. DevOps agent handles this automatically.
